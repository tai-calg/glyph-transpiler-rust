# Compact Glyph syntax

Glyphは、短縮表記を従来文法へ展開してから解析する。既存文法はそのまま使用できる。

```glyph
*S(v,t:f,r:u)
>decode(*S):S|E
  BAD>>Err(BadSensor)
  _>>Ok(S(v,t,r))
```

| 短縮表記 | 展開後 |
|---|---|
| `v,t:f` | `v:f,t:f` |
| 関数引数の `*S` | 積型`S`の全フィールド |
| 型位置の `T|E` | `R<T,E>` |
| ガード行の `condition>>expression` | `condition=>expression` |
| `f` / `d` | `f32` / `f64` |
| `u` / `i` / `b` | `u16` / `i32` / `bool` |

例えば、

```glyph
*S(v,t:f,r:u)
>run(*S):Receipt|E=PIPE
```

は次へ展開される。

```glyph
*S(v:f32,t:f32,r:u16)
>run(v:f32,t:f32,r:u16):R<Receipt,E>=PIPE
```

`|`は文脈で区別する。

- 型シグネチャの最上位にある`T|E`は結果型
- 式位置の`a|b`は論理和
- 直和型宣言の`+A=X|Y`はvariant区切り

式位置の`?`は失敗伝播だけを表す。従来の`R<T,E>`とガード矢印`=>`も互換構文として引き続き使用できる。旧候補だった`T?E`と`T/E`は受理しない。

展開では行数を変えないため、構文エラーの行番号は元の`.glyph`ファイルと一致する。型の一文字短縮は型位置でのみ有効で、同名のユーザー定義型が存在する場合はユーザー定義型を優先する。

ガード関数のフォールバック自体は省略しない。最後の分岐も必ず`_>>expression`と明示し、残りすべての入力を受けることをソース上で可視化する。
